# Milestone v1.0: BankSearchUtils MVP

**Status:** ✅ SHIPPED 2026-01-14
**Phases:** 1-5
**Total Plans:** 7

## Overview

Build a utility that searches the bank by name and withdraws items, starting with core search box infrastructure, then single-item withdrawal, scroll fallback for edge cases, batch operations, and fill-to-capacity functionality. Each phase delivers a complete, testable capability.

## Phases

### Phase 1: Core Search Infrastructure

**Goal**: Implement bank search box interaction - open search, type item name, clear search
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:
- [x] 01-01: Bank Button API Discovery and openSearch()
- [x] 01-02: Search Typing and Clearing

**Details:**
- Created BankSearchUtils.java following existing *Utils patterns
- Used keyboard shortcut (backspace) to activate search instead of button click
- Discovered DialogueType.TEXT_SEARCH and ITEM_SEARCH indicate active search
- PhysicalKey.BACK used instead of ESCAPE (not available in OSMB)

### Phase 2: Single Item Withdrawal

**Goal**: Search for item by name, locate it in results, withdraw specified amount
**Depends on**: Phase 1
**Plans**: 1 plan

Plans:
- [x] 02-01: Single Item Withdrawal Implementation

**Details:**
- searchAndWithdraw(script, itemId, amount) method implemented
- keepSearchOpen parameter added for batch operation efficiency
- Verifies item exists in filtered results before withdrawal

### Phase 3: Scroll Fallback

**Goal**: Scroll through bank contents when search doesn't find item (edge case handling)
**Depends on**: Phase 2
**Plans**: 2 plans

Plans:
- [x] 03-01: Scroll Button Detection (sprite-based)
- [x] 03-02: Scroll Fallback Integration

**Details:**
- Research discovered PixelAnalyzer.findSubImages() doesn't exist
- Implemented sprite-based scroll detection using SpriteManager + ImageAnalyzer
- Sprite IDs: 773 (up), 788 (down), 789/791 (scrollbar)
- Integrated scroll fallback into searchAndWithdraw() as default behavior
- Max 30 scroll iterations safeguard prevents infinite loops

### Phase 4: Batch Withdrawal

**Goal**: Withdraw a list of items with amounts in sequence, handling partial failures
**Depends on**: Phase 2
**Plans**: 1 plan

Plans:
- [x] 04-01: Batch Withdrawal Implementation

**Details:**
- WithdrawalRequest POJO for itemId/amount pairs
- BatchWithdrawalResult tracks successful/failed withdrawals
- withdrawBatch() with List and varargs overloads
- Continues processing on partial failure

### Phase 5: Fill Inventory

**Goal**: Fill all remaining inventory slots with a searched item
**Depends on**: Phase 2
**Plans**: 1 plan

Plans:
- [x] 05-01: Fill Inventory Implementation

**Details:**
- searchAndFillInventory() method implemented
- Returns int (slots filled) matching BankingUtils pattern
- Search-first with scroll fallback for reliability
- Clears search after withdrawal

## Milestone Summary

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Keyboard shortcut over button click | No direct API for BankButtonType clicks | ✓ Good |
| PhysicalKey.BACK over ESCAPE | OSMB doesn't have ESCAPE constant | ✓ Good |
| Sprite-based scroll detection | findSubImages() doesn't exist | ✓ Good |
| Continue batch on partial failure | Callers handle partial results | ✓ Good |
| keepSearchOpen for batch | Efficiency optimization | ✓ Good |

**Issues Resolved:**

- Bank button interaction - solved via keyboard shortcut
- ESCAPE key - solved via BACK key (mobile equivalent)
- Scroll detection - solved via sprite + ImageAnalyzer

**Issues Deferred:**

None

**Technical Debt Incurred:**

None
